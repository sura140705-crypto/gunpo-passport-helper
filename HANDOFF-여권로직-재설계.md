# 여권발급신청서 작성 도우미 — 로직 재설계 인계 문서

> **주의**: 이 문서에는 실제 개인정보나 예시용 식별번호·전화번호·실명이 **포함되어 있지 않습니다.**
> 정부 서식의 "입력 항목 구조"와 프로그램 로직만 설명합니다.
>
> **목적**: 이 문서를 다른 AI에게 그대로 전달하세요. 현재는 "단계 선형 흐름 + 단순 조건" 로직인데,
> 실제 여권 발급 규칙(종류·연령·병역·대리·처리기한 등)이 훨씬 복잡해 재설계가 필요합니다.
> 아래는 (A) 현재 구현, (B) 코드에 끼워 넣을 스키마 명세, (C) 새 로직이 다뤄야 할 도메인 규칙/공백 입니다.
> 새 AI는 (C)를 **외교부 여권 규정(여권법·시행령·시행규칙, passport.go.kr 등)** 기준으로 조사·검증한 뒤,
> (B) 스키마에 맞춰 결정 로직을 산출해 주세요.

---

## 0. 제품 개요 / 불변 제약 (반드시 유지)

- **단일 자체완결 HTML** 파일. 외부 리소스 **0**(CDN·폰트·서버 없음).
- **저장/전송 0**: 입력값(작성 내용)은 브라우저 메모리에서만 처리되고 저장·전송되지 않으며 새로고침 시 사라집니다. 화면·인쇄물에만 사용.
- **화면 구성**: 넓은 화면은 2단 — 한쪽=실제 여권서식 실시간 미리보기, 다른쪽=단계별 "작성 코치". 모바일은 코치만 표시([미리보기] 모달로 서식 확인). 좌우 위치는 토글 버튼으로 전환.
- **대상**: 시청 민원실 키오스크 + 어르신 스마트폰. → 큰 글씨·쉬운 말·"잘 모르겠어요" 배려.
- **성격**: 작성 **보조·미리보기**용이며 접수 가능 여부는 창구 직원 확인이 최종. 애매한 항목은 "직원에게 여쭤볼 항목"으로 미룰 수 있어야 함.

## 1. 코드 구조 (데이터 구동)

로직은 **세 덩어리**로 데이터 구동됩니다. 새 로직은 이 형식에 맞춰 돌려주면 그대로 교체 가능합니다.

| 구성요소 | 역할 |
|---|---|
| `createEmptyState()` | 상태 초기값(입력 항목 키 + 스마트 기본값) |
| `STEP_DEFS[]` | **단계 흐름 정의**(선언적). 각 단계가 무엇을 묻고, 어떤 조건/경고/필수인지 |
| `SUMMARY_ROWS[]` | 마지막 요약에 보여줄 항목·라벨·조건·표시포맷 |
| `CO` 맵 + `renderForm()` | **좌측 실제 서식** 좌표(체크박스·격자칸·자유텍스트·서명란)에 값 얹기 |
| `getStepFieldSet / stepRequired / tryAdvance / updateWarning` | 단계별 항목집합(하이라이트), 필수 항목, 다음 단계 진행 게이팅 |
| `renderStep()` | STEP_DEFS 한 단계를 패널 HTML로 렌더 |

**핵심 계약(중요)**: 상태의 **항목 키(field 이름)** 가 STEP_DEFS · SUMMARY_ROWS · CO(좌측 서식 좌표) 세 곳의 공통 키입니다.
→ **새 항목을 추가하면**, 좌측 서식에 그리려면 `CO`에 좌표를 추가해야 합니다(안 하면 코치/요약에만 보이고 서식엔 안 찍힘). 좌측 서식 좌표·렌더는 완성돼 있으니 **키 이름을 임의로 바꾸지 말 것**.

## 2. 현재 입력 항목(상태 키) 구조

값은 넣지 않고 **키·의미·허용값(옵션)** 만 정리합니다.

| 그룹 | 키 | 의미 | 허용값/형식 |
|---|---|---|---|
| 여권 선택 | `kind` | 여권 종류 | 일반 / 관용 / 외교관 / 긴급 / 여행증명서 |
|  | `travelType` | (여행증명서일 때만) 구분 | 왕복 / 편도 |
|  | `pages` | 면수(속지 수) | 58면 / 26면 |
|  | `period` | 유효기간 | 10년 / 단수(1년) / 잔여기간 / 5년 / 5년 미만 |
| 신청인 | `nameKor` | 한글 성명 | 텍스트 |
|  | `jumin` | 주민등록번호 | 숫자 13자리(표시는 앞6-뒤7 형식) |
|  | `phone` | 본인 연락처 | 휴대전화(자동 하이픈) |
| 긴급연락처 | `emgName` / `emgRel` / `emgPhone` | 성명 / 관계 / 전화 | 본인이 아닌 사람의 연락처 |
| 추가 기재 | `hasPassport` | 기존 여권 보유 | 있음 / 없음 / 모름 |

|  | `regBase` | 등록기준지 | 텍스트(담당자 요청 시) |
| 선택 기재 | `spouseRoman` | 배우자 로마자 성 | 영문 대문자 |

스마트 기본값: `kind=일반, pages=58면, period=10년` 을 미리 선택(그대로 [다음] 가능).

## 3. 현재 단계 흐름 (9단계, 선형)

| # | 제목(section) | 수집/동작 | 필수 | 조건·경고·비고 |
|---|---|---|---|---|
| 1 | 사전 체크리스트(시작) | 준비물 안내(체크리스트만, 입력 없음) | – | 입력 없음 |
| 2 | 여권 선택(여권) | `kind`, `period`, `pages`, (조건)`travelType` | kind·period·pages | `travelType`은 `kind`가 여행증명서일 때만 노출·활성 |
| 3 | 신청인 정보(신청인) | `nameKor`, `jumin`, `phone` | 3개 모두 | 주민/전화 자동 하이픈 |
| 4 | 기존 여권(로마자) | `hasPassport`(있음/없음/모름) | hasPassport | 로마자 분기용으로 먼저 물음 |
| 5 | 로마자 성명(로마자) | `romanSur`, `romanGiven` | 2개 | 한글이름 기반 **권장 예시 제안(자동확정X)**. 기존여권 있음이면 "기존 여권과 철자 일치" 강경고 |
| 6 | 긴급 연락처(긴급연락처) | `emgName`, `emgRel`, `emgPhone` | 3개 | 긴급번호==본인번호면 경고 |
| 7 | 추가 기재(추가기재) | `regBase`, `spouseRoman`, `braille` | 없음(전부 선택) | "해당 시에만". 대부분 비워 둠 |
| 8 | 수령 방법(수령) | `mailService`, (조건)`mailAddr` | 없음 | 우편 희망 시 주소 필수 + 요금 안내 |
| 9 | 확인·완료(완료) | 요약 + "직원에게 여쭤볼 항목" + 인쇄/PDF | – | 서명·날인은 인쇄 후 수기 |

**진행/게이팅**:
- `stepRequired(def)` = 그 단계의 필수 항목(조건 충족한 것만).
- `tryAdvance()` = [다음] 클릭 시 필수 미입력이면 차단·경고·포커스. 단 **"🤔 잘 모르겠어요"** 를 누르면 그 단계를 `unsure`로 표시하고 필수라도 통과 → 마지막 요약의 "직원에게 여쭤볼 항목"으로 모음.
- `updateWarning()` = 필수 미입력이 있으면 "진행은 가능하나 비었다" 부드러운 안내(차단 아님).

## 4. 현재 "인코딩된" 규칙 (전부 — 매우 얕음)

지금 코드가 실제로 하는 규칙은 이게 전부입니다:

1. **기본값 제안**: 일반·10년·58면. (안내 문구: "표시 없으면 이대로 발급")
2. **여행증명서 → 왕복/편도** 하위 선택 노출(그 외 종류엔 숨김·해제).
3. **기간 옵션에 연령 힌트 텍스트만** 존재(로직 아님): "10년=성인 대부분 / 5년=일정 연령대 / 5년 미만=저연령". → **실제 나이로 자동 제한·추천하지 않음.**
4. **로마자**: 한글→권장 예시(성씨 관용표기 + 음절 로마자화) 제안만, 자동 확정 안 함. 기존여권 있음 → 일치 강경고.
5. **긴급연락처가 본인 번호와 같으면** 경고.
6. **우편 희망 → 주소 필수 + 요금 별도 안내**.
7. **번호 항목 자동 하이픈** 정리.

**즉, "종류·나이·병역·대리 여부"에 따라 흐름/필수/가능여부가 바뀌는 로직은 전혀 없음.** (재설계 핵심)

## 5. STEP_DEFS 스키마 명세 (새 로직을 이 형식으로)

한 단계 객체:
```js
{
  title:"단계 제목", section:"진행바 짧은 그룹명",
  question:"큰 질문 문구", why:"쉬운 설명(💡 박스)",
  checklist:[ "문자열", ... ],              // 안내용 체크리스트(입력 없음)
  terms:[ {term:"용어", desc:"쉬운 풀이"} ],  // '쉬운 말 풀이' 접이식
  suggestRoman:true,                          // 로마자 권장예시 블록(특수)
  groups:[                                    // 큰 선택 버튼(라디오형)
    { field:"kind", label:"...", req:true|false,
      cond:function(d){ return 조건; },        // 참일 때만 노출
      options:[ {v:"저장값", label:"표시값?", rec:true?, desc:"설명?"} ] }
  ],
  inputs:[                                    // 타이핑 입력
    { field:"...", label:"...", req:bool, type:"text|tel",
      mask:"jumin|phone"?, inputmode?, maxlength?, ph:"placeholder", hint:"도움말", upper:true? }
  ],
  condInputs:[ { ...inputs와 동일 + cond:fn } ],
  addrInput:{ field:"mailAddr", label:"...", req:bool, cond:fn },   // 구조화 주소 입력
  warn:{ when:function(d){...}, msg:"⚠ 강경고 문구" },              // 조건부 빨강 경고
  note:{ when:function(d){...}, msg:"노랑 안내 문구" },              // 조건부 안내
  summary:true                                // 이 단계는 요약/완료 화면
}
```
- `field`(항목 키) = 상태 키 = `CO`(좌측 서식) 키. **일관 유지 필수.**
- `options[].v` = 실제 저장값(CO 좌표 키와 일치해야 좌측 체크가 맞음). `label`은 표시용.
- 렌더 순서: question → why → terms → checklist → warn → (unsure표시) → suggestRoman → inputs → condInputs → **groups** → note → addrInput → "잘 모르겠어요" 버튼.

**새 로직이 필요로 할 훅(현재 없음 — 신설 권장)**:
- **단계-레벨 `cond`**(단계 자체를 조건부 스킵). 현재는 모든 단계가 항상 표시됨. 종류·나이별로 단계를 건너뛰려면 필요.
- `def.validate(d)` 훅은 호출부가 이미 있음(`tryAdvance`가 있으면 실행) → 커스텀 검증 메시지 반환 가능(현재 미사용).
- **파생 값 계산 헬퍼**: 주민번호→생년월일→만나이 등. 현재 없음.

## 6. ★ 새 로직이 다뤄야 할 도메인 규칙/공백 (연구 대상)

> 아래는 **현재 무시하고 있는** 결정 요인들입니다. 각 항목을 최신 여권 규정으로 사실 확인한 뒤,
> "입력 → 가능여부 / 필수항목 / 처리기한 / 흐름분기"로 규칙화해 주세요. (규정은 개정되므로 반드시 최신 확인)

### 6-1. 여권 종류(kind)별 분기
- **관용·외교관**: 개인이 창구에서 바로 신청하는 대상이 아님(소속 기관·외교부 경유). → 선택 시 "창구/소속기관 문의"로 안내하고 일반 흐름과 분리할지?
- **긴급여권**: 사유·처리·수수료·유효기간(단수 성격) 규칙이 다름. 면수 선택이 의미 없을 수 있음.
- **여행증명서**: 여권과 별개 서식/절차. 왕복·편도 외 요건.
- → 종류별로 "면수 선택 노출 여부, 기간 옵션, 필요 서류, 처리기한, 신청 채널"이 달라짐.

### 6-2. 연령 기반 규칙 (주민번호에서 만나이 계산)
- 유효기간 가능 범위가 나이로 갈림(성인 장기 / 일정 연령대 / 저연령 단수 등) — 현재는 힌트 텍스트뿐.
- **미성년자**: 법정대리인 동의·서명, 대리 신청 요건, 필요 서류(가족관계 증빙 등).
- → 나이를 계산해 **기간 옵션을 필터/추천**하고 미성년 분기(대리인 단계)를 띄워야 함.

### 6-3. 병역 관계 (해당 시)
- 특정 연령대 남성의 국외여행허가·단수/제한 등. → 성별·나이로 병역 관련 안내/분기 필요 여부 확인.

### 6-4. 대리(위임) 신청
- 본인 방문이 원칙이나 예외(미성년·질병 등) 대리 가능. 대리 시 위임장·대리인 신분증·관계 증빙 등 추가 항목·단계 필요.
- → "본인/대리" 선택 단계 + 대리인 정보 항목 신설 검토.

### 6-5. 신청 성격 (신규 / 재발급 / 기재변경 / 분실재발급)
- `hasPassport`를 지금은 로마자 일치용으로만 사용. 실제로는 신청 성격에 따라 서류·수수료·로마자 변경 가능여부가 다름. 분실은 별도 절차.

### 6-6. 로마자성명 규칙
- 최초/변경, 가족 표기 일치 권고, 변경 제한 사유, 표기 규정 등. (현재는 관용표기 제안 + 기존여권 일치 경고 수준)

### 6-7. 처리기한/수수료
- 종류·면수·기간·긴급여부별로 수수료와 처리 소요일이 다름. 현재는 전부 "창구 안내"로 미룸.
- → 규칙표로 정리 시 요약 단계에서 예상 수수료·기한 안내 가능(지자체/시점별 상이 → 데이터 분리 권장).

### 6-8. "직원 확인 필요" 경계
- 자동 판정이 애매/위험한 항목(관용·외교관·병역·대리·분실 등)은 억지 자동화 대신 "직원 확인" 흐름으로 유도(기존 `unsure` 메커니즘 활용).

## 7. 권장 재설계 방향 (제안)

현재 "선형 STEP_DEFS"에 **결정 규칙 계층(rules)** 을 분리 도입:

1. **파생 상태 계산** `derive(data)` → `{ ageYears, isMinor, sex, applyType(신규/재발급/…), isSelfApply, … }`.
2. **규칙 테이블(데이터)**: 종류·나이·대리 등 입력 조합 → `{ 허용 period 옵션, 면수 노출여부, 추가 필요단계, 처리기한, 수수료, 경고, "직원확인" 플래그 }`.
3. **동적 흐름**: 단계-레벨 `cond`로 해당 없는 단계 스킵, 미성년/대리 시 단계 삽입. 옵션 목록도 규칙 결과로 필터.
4. **표시(STEP_DEFS/렌더)는 그대로 재사용** — 규칙 결과를 `groups.options`/`req`/`warn`에 주입.
5. 규정은 **출처·버전을 주석**으로 남기고, 시점/지자체 의존 값(수수료·기한)은 설정으로 분리.

**돌려줄 산출물 형태**(택1):
- (a) 위 스키마에 맞춘 새 `STEP_DEFS` + `createEmptyState` + 규칙 헬퍼(순수 JS, 외부 의존 0), 또는
- (b) 규칙 명세(입력→결과 표)만 — 이쪽에서 코드로 옮김.

어느 쪽이든 **항목 키 계약(§2)과 §0 제약(단일 HTML·무저장·무외부의존)을 지켜** 주세요.
